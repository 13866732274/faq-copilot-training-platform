## 1. 项目概述

### 1.1 项目背景

公司使用微E聊（VEL）软件管理多个微信客服账号的对话记录。为加速新入职咨询员的培训效率，需要将历史优秀对话（极佳评级）转化为可反复训练的模拟训练题库，使咨询员通过"训练"方式快速掌握话术技巧。

### 1.2 系统定位

**咨询话术模拟训练系统** —— 基于真实历史优秀对话记录，为新入职咨询员提供仿微信聊天界面的训练式话术训练平台。

### 1.3 核心价值

| 价值点 | 说明 |
|--------|------|
| 降低培训成本 | 咨询员可自主反复训练，减少资深员工带教时间 |
| 标准化话术 | 基于真实优秀案例，统一话术标准 |
| 量化培训进度 | 管理员可查看每人练习进度和答题质量 |
| 持续积累 | 案例库可持续扩充，优秀案例越积越多 |

## 2. 需求规格说明

### 2.1 用户角色

| 角色 | 权限 | 说明 |
|------|------|------|
| **超级管理员** | 全部权限 | 系统设置、用户管理、案例库管理、数据统计 |
| **管理员/主管** | 案例库管理 + 查看统计 | 导入题目、编辑题库、查看所有咨询员训练记录、针对性讲解 |
| **新入职咨询员** | 训练 + 查看自己记录 | 选择案例练习、查看历史训练记录和标准答案对比 |

### 2.2 功能需求清单

#### F1 - 用户管理模块

| 编号 | 功能 | 优先级 | 说明 |
|------|------|--------|------|
| F1.1 | 用户注册/登录 | P0 | 支持用户名+密码登录，JWT Token鉴权 |
| F1.2 | 角色权限管理 | P0 | 超级管理员、管理员、咨询员三种角色 |
| F1.3 | 用户列表管理 | P0 | 管理员可查看、新增、禁用用户 |
| F1.4 | 个人信息修改 | P1 | 修改密码、头像、昵称 |

#### F2 - 案例库管理模块

| 编号 | 功能 | 优先级 | 说明 |
|------|------|--------|------|
| F2.1 | HTML文件导入 | P0 | 上传微E聊导出的HTML文件，系统自动解析为对话题目 |
| F2.2 | 导入预览 | P0 | 导入后预览解析结果，确认患者/咨询师角色识别是否正确 |
| F2.3 | 案例列表 | P0 | 查看所有已导入题目，支持搜索、筛选、分页 |
| F2.4 | 题目编辑 | P1 | 手动修改解析后的消息内容、角色标记 |
| F2.5 | 案例分类/标签 | P1 | 按科室、疾病类型、难度等分类 |
| F2.6 | 题目启用/停用 | P0 | 控制题目是否对咨询员可见 |
| F2.7 | 题目删除 | P1 | 软删除，可恢复 |
| F2.8 | 批量导入 | P1 | 支持一次上传多个HTML文件批量导入 |

#### F3 - 模拟训练模块（核心）

| 编号 | 功能 | 优先级 | 说明 |
|------|------|--------|------|
| F3.1 | 案例选择 | P0 | 咨询员可浏览案例列表，选择一个案例开始训练 |
| F3.2 | 随机出题 | P1 | 支持随机抽取一个案例开始训练 |
| F3.3 | 模拟对话界面 | P0 | 仿微信聊天界面，左侧显示患者消息，右侧为咨询员回复区 |
| F3.4 | 逐条出题 | P0 | 患者消息逐条展示，咨询员回复后才出现下一条患者消息 |
| F3.5 | 多媒体展示 | P1 | 支持显示文字、图片；语音以播放按钮或转文字形式展示 |
| F3.6 | 提交答卷 | P0 | 完成全部对话后，点击"提交"结束本次练习 |
| F3.7 | 答案对比 | P0 | 提交后展示完整对话：左列为标准答案，右列为咨询员回复，逐条对照 |
| F3.8 | 中途保存 | P2 | 可中途退出，下次继续（进度保存） |

#### F4 - 学习记录与统计模块

| 编号 | 功能 | 优先级 | 说明 |
|------|------|--------|------|
| F4.1 | 个人训练记录 | P0 | 咨询员可查看自己的历史训练记录 |
| F4.2 | 训练详情回顾 | P0 | 点击某次练习记录，可重新查看答案对比 |
| F4.3 | 管理员统计面板 | P0 | 查看所有咨询员的练习次数、已完成案例数、最后练习时间 |
| F4.4 | 管理员查看详情 | P0 | 管理员可查看任意咨询员的任意一次训练详情和对比 |
| F4.5 | 管理员点评/讲解 | P1 | 管理员可针对某次练习添加点评/评语，咨询员可查看 |
| F4.6 | 数据导出 | P2 | 导出统计数据为Excel |

### 2.3 非功能需求

| 需求 | 说明 |
|------|------|
| 并发支持 | 支持 50 人同时在线使用 |
| 响应时间 | 页面加载 < 2秒，接口响应 < 500ms |
| 数据安全 | 密码加密存储（bcrypt），Token鉴权，接口权限控制 |
| 浏览器兼容 | Chrome 90+、Edge 90+（主要面向PC端使用） |
| 可维护性 | 代码规范、接口文档完善（Swagger自动生成） |

## 3. 技术架构

### 3.1 技术栈选型

| 层面 | 技术 | 当前实际版本/要求 | 说明 |
|------|------|-------------------|------|
| **前端框架** | Vue 3 + TypeScript | Vue `^3.5.25` + TypeScript `~5.9.3` | 当前实际运行版本，主流稳定 |
| **前端构建** | Vite | `^7.3.1` | 本地开发与生产构建 |
| **前端 UI** | Element Plus | `^2.13.3` | 管理后台与咨询员端统一组件体系 |
| **前端状态管理** | Pinia | `^3.0.4` | Vue 3 官方推荐状态管理 |
| **前端路由** | Vue Router | `^4.6.4` | 管理端/咨询员端双路由体系 |
| **前端 HTTP** | Axios | `^1.13.6` | REST API 请求；流式接口使用原生 `fetch` |
| **图表能力** | ECharts | `^6.0.0` | 统计看板与运营报表 |
| **后端框架** | FastAPI | `>=0.110.0` | 异步 API 框架 |
| **ASGI 服务** | Uvicorn | `>=0.27.0` | 后端运行入口 |
| **后端 ORM** | SQLAlchemy 2.x | `>=2.0.25` | 异步 ORM |
| **数据库迁移** | Alembic | `>=1.13.0` | 数据库版本管理 |
| **数据库** | MariaDB / MySQL | MySQL 8.0+ / MariaDB 兼容 | 当前项目按 MySQL 语义设计 |
| **数据库驱动** | asyncmy + pymysql | `>=0.2.9` / `>=1.1.0` | 异步主驱动 + 兼容支持 |
| **缓存 / 队列** | Redis | 7.x | Celery broker/backend 与缓存能力 |
| **异步任务** | Celery | 与 Redis 配套 | FAQ 处理、增量更新、分布式 embedding |
| **HTML 解析** | BeautifulSoup4 + lxml | `>=4.12.0` | 常规解析 + 大文件流式解析 |
| **CSV/Excel 解析** | 自研解析器 + openpyxl 兼容 | - | 支持 HTML / CSV / XLSX |
| **聚类 / 数值计算** | scikit-learn + NumPy | `>=1.5.0` / `>=2.0.0` | FAQ 聚类与向量计算 |
| **认证安全** | python-jose + passlib[bcrypt] | `>=3.3.0` / `>=1.7.4` | JWT + bcrypt |
| **AI 大模型** | DashScope / 通义千问 | 管道+quality 模式：`qwen-plus`；auto/fast/balanced：`qwen-turbo` | 高置信直通优先，跳过 LLM |
| **Embedding** | DashScope `text-embedding-v3` | 1024 维 | 问题语义向量 |
| **Web 服务器** | Nginx | 1.24+ | 反向代理与静态资源服务 |
| **进程管理** | systemd | - | 当前实际使用 `systemd` 管理后端/Celery |
| **容器化（可选）** | Docker + Docker Compose | 24+ | 可选，不是当前主部署方式 |

### 5.1 微E聊 HTML 导出格式分析

根据实际导出文件分析，HTML 文件结构如下：

```html

  </div>

  </div>

  <!-- 咨询师消息（右对齐），含 text-right 和 talkdetailedduihua1-m 类名 -->
```

### 5.2 解析规则

| 特征 | 患者消息 | 咨询师消息 |
|------|----------|------------|
| CSS类名 | `text-left` | `text-right` |
| 背景类名 | `talkdetailedduihua-m`（白色背景） | `talkdetailedduihua1-m`（绿色背景） |
| 昵称位置 | 昵称在前，时间在`<span>`中 | 时间在`<span>`中在前，昵称在后 |
| 内容前缀 | 可能包含`[同步消息]` | 可能包含`[同步消息]` |
| 图片消息 | `<img>` 标签 | `<img>` 标签 |
| 语音消息 | 含音频相关标签或`[语音]`标记 | 含音频相关标签或`[语音]`标记 |

#### 12.26.2 “AI 精炼”到底在做什么（是否按需求清洗无用信息）

是按你的目标做的，且有明确规则与代码落点：

- 清洗规则由 `EXTRACT_QA_SYSTEM_PROMPT` 定义（文件：`backend/app/services/faq_llm.py`）
  - 只提取实质咨询问题（地址、费用、流程、方案等）
  - 过滤寒暄、单字回复、无意义闲聊、非文本内容
  - 连续追问同一问题会合并
  - 咨询师多条连续回答会合并为完整答复
- 模型返回严格 JSON（`response_format=json_object`），解析失败自动丢弃该轮
- 每条抽取结果保留消息序号映射（`question_msg_ids` / `answer_msg_ids`），可追溯原始对话位置

结论：当前“清洗无用信息”是 **LLM 规则抽取 + 结构化约束**，不是黑盒自由生成。

#### 12.26.3 数据聚合到底用百炼还是算法（精准说明）

当前是你要求的 **混合方案（Hybrid）**，分工如下：

1. **百炼大模型（Dashscope / qwen3-max）负责语义理解层**
   - 从原始对话抽取问答对（`extract_qa_pairs`）
   - 对每个聚类生成标题/摘要/代表问题/最佳答案（`refine_cluster`）
2. **算法负责稳定分组层**
   - 问题向量：Dashscope `text-embedding-v3`（1024 维）
   - 聚类算法：`AgglomerativeClustering(metric="cosine", linkage="average")`
   - 阈值：`distance_threshold = 0.35`（定义于 `faq_pipeline.py`）
3. **检索层**
   - 语义搜索按余弦相似度排序（向量归一化后点积）
   - Copilot 在 Top-K FAQ 上再次调用 LLM 组织推荐话术

所以：**“分哪一类”主要靠向量 + 聚类算法；“这类叫什么、最佳答复是什么”主要靠百炼大模型。**

> **背景：** 前 5 次 FAQ 处理任务全部在 `step_refine_and_store` 阶段崩溃，根因是 message_id 外键约束失败。

**根因分析：**

管线调用 LLM 提取 Q&A 对时，对话文本以 `[sequence][role] content` 格式传入。LLM 返回的 `question_msg_ids` / `answer_msg_ids` 是消息**序号**（如 10、47、52），但代码直接将序号作为 `message_id` 写入 `faq_questions` 和 `faq_answers` 表。由于 `messages` 表的 `id`（自增主键）与 `sequence`（对话内序号）不一致（例如 quiz_id=14 的第 1 条消息 id=724、sequence=1），外键约束 `faq_answers_ibfk_4` 一定会失败。

**修复方案：**

1. **序号→真实 ID 映射**
   - `_load_conversations` 现在返回 `(quiz, messages, seq_to_real_id)` 三元组
   - 新增 `_resolve_msg_ids(seq_ids, seq_to_real)` 将 LLM 返回的序号批量转为数据库真实 ID
   - 无法映射的序号自动跳过（容错 LLM 幻觉）

2. **逐 Cluster 提交**
   - `step_refine_and_store` 改为每个 cluster 单独 `commit`
   - 单个 cluster 写入失败 → `rollback` + 跳过，不影响其余 cluster
   - 新增日志 `Cluster 3/15 saved (4 pairs)` 方便定位

3. **清理旧数据**
   - 每次管线开始 refining 前先删除该租户的旧 FAQ 数据（`_clean_previous_faq_data`）
   - 防止重试任务产生重复数据

4. **异常处理安全化**
   - `run_pipeline` 异常分支先 `rollback` 再更新 task 状态
   - 使用预存的 `task_id` 变量避免在 session 过期后访问 `task.id`（此前该二次异常被吞掉，导致真正的错误信息丢失）

**修复后效果：**

- 任务 #6：9 个对话 / 492 条消息 → 提取 22 对 Q&A → 生成 15 个聚类 → **总耗时 2 分 50 秒**
- 分类：基础信息(5)、治疗方案(4)、就诊流程(2)、费用咨询(2)、症状咨询(1)、预约相关(1)

**涉及文件：**

- `backend/app/services/faq_pipeline.py` — 核心修复
- `backend/app/api/faq.py` — 异常处理

#### 12.26.13 Copilot SSE 流式输出 + 查询日志 + 提速优化（2026-03-14 已实施）

> **背景：** AI 问答助手响应 5-10 秒，无查询记录，无法追踪用户使用情况。

**问题诊断：**

| 瓶颈 | 原因 | 影响 |
|------|------|------|
| 非流式输出 | 等 LLM 全部生成完才返回 | 用户空等 5-10s 无反馈 |

| max_tokens 过高 | 1024 tokens | 超出实际需要，浪费时间 |

   - 所有 API 调用（Embedding、LLM、Copilot）共享连接池
   - 省去每次 TLS 握手 ~200ms

3. **Copilot 专用快速模型**
   - 新增配置 `dashscope_copilot_model = "qwen-turbo"`
   - Copilot 场景用 `qwen-turbo`（更快更便宜），管线继续用 `qwen3-max`（更精准）
   - `max_tokens` 从 1024 降至 512

4. **查询日志记录**
   - 新表 `faq_copilot_logs`（迁移：`20260314_01`）
   - 记录字段：user_id、mode、query、reply、confidence、matched_count、latency_ms
   - 所有模式（copilot / copilot_stream / search）均自动记录
   - 管理员可在"查询日志"页面查看所有使用记录

5. **查询日志管理页面**
   - 路由：`/admin/faq/copilot-logs`
   - 菜单：FAQ 知识库 → 查询日志
   - 显示：用户、模式、查询内容、AI 回复、匹配数、耗时、时间
   - 耗时标签颜色：<1.5s 绿色、1.5-3s 黄色、>3s 红色

**提速效果：**

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| 首 token 可见 | 5-10s | **~0.5s** |
| 完整回复 | 5-10s | **~1-2s** |

## 12.27 整站系统优化（运营视角全面升级）

### 12.27.1 优化目标

从运营管理实际使用角度出发，对"数据导入→清洗提取→知识库生成→Copilot 问答"全链路进行系统性优化，确保大量对话导入时数据质量可控、角色识别可靠、FAQ 生成精准。

### 12.27.2 Copilot 自动/极速/平衡/高质量路由机制（详解）

#### 路由判定逻辑

| 模式 | 触发条件 | 实际模型 | top_k | max_tokens | 典型延迟 |
|------|----------|----------|-------|------------|----------|
| **auto** | 默认选择 | 见下方路由规则 | — | — | — |
| **fast** | 手动选择 或 auto 判定短问句 | qwen-turbo | 5 | 384 | ~1-2s |
| **balanced** | 手动选择 或 auto 默认 | qwen-plus | 6 | 512 | ~2-3s |
| **quality** | 手动选择 或 auto 命中风险词 | qwen3-max | 8 | 768 | ~3-5s |

#### auto 自动路由规则（`_resolve_effective_quality_mode`）

```
1. 如果用户明确选择了 fast/balanced/quality → 直接使用该模式
2. auto 模式下：
   a. 查询长度 ≤ 4 字且不含风险关键词 → fast（短问句极速响应）
   b. 查询包含风险关键词 → quality（需要最精准回答）
   c. 其余情况 → balanced（日常推荐）
```

#### 风险关键词（可配置化）

- **默认关键词**：费用、价格、多少钱、收费、医保、报销、疗程、疗效、副作用、禁忌、复发、手术、住院、风险、药物、孕妇、儿童、老人
- **配置位置**：`system_settings.copilot_risk_keywords` 字段（逗号分隔）
- **每个租户可独立配置**不同的风险关键词集合
- **加载逻辑**：Copilot 每次请求时从 DB 读取当前租户的关键词配置

#### 模型选择依据

| 模型 | 能力 | 适用场景 |
|------|------|----------|
| **qwen-turbo** | 速度最快，理解力一般 | 简短问题（地址在哪？） |
| **qwen-plus** | 速度与质量平衡 | 日常问答（挂号流程、预约方式） |
| **qwen3-max** | 最强理解力，速度稍慢 | 复杂/敏感问题（费用、医保、手术风险） |

#### 可靠性评估

✅ **合理**：三级模型分层与医疗咨询场景匹配，费用/手术等高敏感问题需要最精准的回答  
✅ **可配置**：风险关键词和聚类阈值均可通过系统设置调整，不需改代码  
✅ **可追踪**：每次请求记录 `requested_mode → effective_mode` 和 `route_reason`  
⚠️ **注意**：auto 模式依赖关键词匹配，未覆盖的敏感词会走 balanced；建议运营定期补充关键词

### 12.27.3 患者/咨询师角色识别增强

#### 当前识别机制

```
HTML 结构：div.conversation > div.nickname-m > (text-right → counselor, 其他 → patient)
```

角色完全由 HTML 中 `div.nickname-m` 的 CSS class 决定：
- 含 `text-right` → `counselor`（咨询师/客服）
- 不含 `text-right` → `patient`（患者/访客）

#### 本次增强：质量报告（RoleQualityReport）

每次 HTML 上传解析后，自动生成 `RoleQualityReport`：

| 指标 | 说明 |
|------|------|
| `patient_count` / `counselor_count` | 各角色消息数量 |
| `text_count` / `image_count` / `audio_count` | 各类型消息数量 |
| `role_alternation_rate` | 角色交替率（0-1），越高说明对话结构越正常 |
| `unique_patient_names` | 去重后的患者昵称列表 |
| `unique_counselor_names` | 去重后的咨询师昵称列表 |
| `name_collision` | 同一昵称是否同时出现在两个角色（严重警告） |
| `quality_score` | 综合质量评分（0.0-1.0） |
| `warnings` | 具体警告列表 |

#### 质量评分扣分规则

| 条件 | 扣分 |
|------|------|
| 姓名冲突（name_collision=true） | -0.3 |
| 缺少任一角色（patient=0 或 counselor=0） | -0.4 |
| 文本消息占比 < 30% | -0.2 |
| 角色交替率 < 15%（且消息数≥5） | -0.1 |

#### 接口变更

`POST /api/v1/quizzes/upload` 响应新增 `quality_report` 字段。

#### 大量导入时的注意事项

1. **角色识别依赖 HTML 结构**：目前仅支持特定的 HTML 格式（微信/企微导出格式），若来源不同的平台需适配 `html_parser.py`
2. **质量报告帮助预检**：导入前通过 `quality_report.warnings` 快速发现问题文件
3. **文件大小限制**：单文件 ≤ 20MB，超大文件建议拆分

### 12.27.4 FAQ 管道增量更新

### 12.27.7 批量上传方案分析

#### 当前方案

```
[用户上传 HTML] → [BeautifulSoup 解析] → [预览确认] → [入库]
                                                          ↓
[FAQ 处理任务] → [LLM 提取 Q&A] → [Embedding] → [聚类] → [LLM 精炼] → [写入]
```

#### 评估

| 维度 | 现状 | 评价 |
|------|------|------|
| **格式支持** | 仅 HTML | ⚠️ 局限，后续需支持 Excel/CSV 模板导入 |
| **角色识别** | HTML class 单一判据 + 质量报告 | ✅ 足够可靠（微信导出格式稳定） |
| **数据清洗** | LLM 驱动（qwen3-max 提取 Q&A） | ✅ 精准度高，可追溯到原始消息 |
| **聚类方案** | text-embedding-v3 + 层次聚类 | ✅ 效果好，阈值可配置 |
| **增量能力** | 锁定保护 + 全量重生成 | ⚠️ 可接受，后续可做真增量 |
| **并发处理** | 后台单任务串行 | ⚠️ 几百张表时需队列化 |
| **大文件处理** | 全量读入内存 | ⚠️ 20MB 限制保护，超大需拆分 |

#### 当前方案是否最优？

**对于现阶段（9-50 个对话文件），当前方案已足够可靠**。理由：

1. ✅ LLM 提取比规则匹配更智能（能处理各种提问方式）
2. ✅ 向量聚类效果好（语义相似的问题自动归组）
3. ✅ 每步有日志和监控（失败可重试、有心跳超时保护）

### 12.28.2 真增量更新

#### 机制

```
POST /api/v1/faq/process-incremental
```

1. 查询所有活跃对话的 ID 集合 `A`

### 12.28.3 并行处理优化

#### Embedding 并行化

```python
CONCURRENT_BATCHES = 3  # 同时发送 3 个 embedding 批次
batches = [texts[i:i+10] for i in range(0, len(texts), 10)]
for chunk in chunks(batches, CONCURRENT_BATCHES):
    results = await asyncio.gather(*[_embed_batch(b) for b in chunk])
```

**效果**：嵌入阶段提速约 2-3 倍。

#### LLM 提取并行化

```python
EXTRACT_CONCURRENCY = 3  # 同时提取 3 个对话
for chunk in chunks(conversations, EXTRACT_CONCURRENCY):
    results = await asyncio.gather(*[_extract_one(q, m, s) for q, m, s in chunk])
```

**效果**：提取阶段提速约 2-3 倍。

### 12.28.7 P3 实施记录（已完成）

#### 12.28.7.1 智能聚类合并

增量更新时，新提取的聚类不再一律创建新条目，而是先与已有 FAQ 做向量相似度匹配：

```
新聚类 centroid → cosine similarity → 所有已有 cluster embedding
  ├── similarity ≥ 0.82 → 合并到已有 cluster（追加 Q&A、更新计数和质心）
  └── similarity < 0.82 → 创建新 cluster
```

**核心逻辑**：

| 函数 | 职责 |
|------|------|
| `_load_existing_cluster_embeddings()` | 加载租户下所有活跃 cluster 的归一化向量 |
| `_find_best_merge_target()` | 找到最高相似度的已有 cluster，判断是否超过阈值 |
| `_merge_into_existing_cluster()` | 追加 Q&A 记录、更新计数、平均混合质心向量 |

**合并阈值**: `MERGE_SIMILARITY_THRESHOLD = 0.82`（余弦相似度），确保只有语义高度相似的才合并。

**效果**：
- 避免增量更新后出现大量重复 FAQ 条目
- 合并后的 cluster 自动更新 `question_count`、`answer_count`、`embedding_json`
- 新创建的 cluster 也会加入到本轮的 existing 缓存中，支持同批内互相合并

#### 12.28.7.2 分布式 Embedding

当待嵌入文本 ≥ 50 条时，自动尝试通过 Celery 分发 embedding 任务：

```
step_embed()
  ├── len(questions) >= 50 → celery group(embed_batch * N chunks)
  │     ├── Worker 1: embed chunk 0-29
  │     ├── Worker 2: embed chunk 30-59
  │     └── Worker N: embed chunk ...
  └── len(questions) < 50 → 本地 asyncio.gather（原方式）
```

| 组件 | 说明 |
|------|------|
| `faq.embed_batch` Celery task | 接收 texts 列表，返回 embeddings |
| `faq.embed` 队列 | 专用 embedding 队列 |
| `_try_distributed_embed()` | 切分 chunk→group→apply_async→get(timeout=300s) |
| 降级策略 | Celery 不可用时自动 fallback 到本地并行 |

**扩展方式**：
```bash
# 在另一台服务器上启动专用 embedding worker
celery -A app.celery_app:celery_app worker --concurrency=4 --queues=faq.embed -n embed@remote-host
```

#### 12.28.7.3 流式解析（lxml）

为大文件（>2MB）提供 lxml 流式解析方案，显著降低内存占用：

| 条件 | 解析器 | 内存使用 |
|------|--------|----------|
| 文件 ≤ 2MB | BeautifulSoup（html.parser） | 约 3-5 倍文件大小 |
| 文件 > 2MB | lxml etree（C 实现） | 约 1-2 倍文件大小 |

```python
def parse_html_auto(html_content: str) -> ParsedConversation:
    if len(html_content.encode()) > 2 * 1024 * 1024:
        return parse_vel_html_streaming(html_content)
    return parse_vel_html(html_content)
```

**降级保障**：lxml 解析失败或结果为空时，自动 fallback 到 BeautifulSoup。

#### 12.28.7.4 任务优先级队列

基于 Redis 的 Celery 优先级队列，确保增量任务优先执行：

| 任务 | 最大重试 | 退避策略 |

| `embed_batch` | 3 次 | 指数退避 |

- `task_soft_time_limit = 1800s`（30 分钟软超时，触发 SoftTimeLimitExceeded）

#### 12.29.4 VEL → 知识库 → 回复辅助的实际处理链路

经本轮代码审计，当前 FAQ/知识库链路的**实际顺序**为：

```text
VEL HTML / CSV / XLSX 导入
→ 解析对话并写入 quizzes + messages
→ 触发 FAQ 全量/增量处理任务
→ LLM 从对话中抽取问答对
→ 对问题生成 embedding
→ 对问题向量做层次聚类
→ LLM 精炼每个 cluster 的标题/摘要/最佳答案
→ 落库为 faq_clusters / faq_questions / faq_answers
→ 查询时做 query embedding
→ 与 FAQ cluster 向量做余弦相似度搜索
→ 把 top-k FAQ 上下文交给 Copilot 生成推荐回复
```

这意味着当前系统不是“直接拿原始对话去喂模型做即时问答”，而是采用了更适合运营与培训的结构化链路：

1. **先沉淀知识**
2. **再做语义搜索**
3. **最后生成建议回复**

这种做法对于“新入职咨询员培训 + 日常咨询辅助”场景是合理的，因为它更稳定、更可控、更容易积累知识资产。

#### 12.29.5 本轮审计结论

##### 结论总览

| 维度 | 审计结论 |
|------|----------|
| 技术栈 | 采用 2026 年主流且成熟的前后端技术栈，无明显落后组件 |
| 数据处理链路 | VEL 解析 → Q&A 抽取 → embedding → 聚类 → 精炼 → 检索 → Copilot，整体合理 |
| 新入职咨询员培训场景适配度 | 高，尤其适合标准话术沉淀、复盘和快速上手 |
| 微信/商务通辅助场景 | 可用，尤其适合“问题输入 → 快速给建议回复”的半自动辅助 |

| 当前内网可用性 | 已具备长期内网使用条件，不必为了当前阶段额外做 P0 改造 |

##### 具体判断

1. **整体流程设计合理**
   - 当前不是简单的聊天机器人，而是“训练系统 + FAQ 知识库 + Copilot 辅助”的复合架构。
   - 用历史优秀对话做训练语料和知识抽取，方向正确。

2. **技术路线符合 2026 常见最佳实践**
   - 前端：Vue 3 + Vite + TS + Pinia + Element Plus
   - 后端：FastAPI + SQLAlchemy 2.x + Alembic
   - AI：DashScope Qwen + `text-embedding-v3`
   - FAQ：LLM 抽取 + embedding + 聚类 + 检索增强生成（RAG 风格）

3. **当前规模下无需过度升级架构**
   - FAQ 检索当前采用“全量 cluster 向量扫描”的简单方式。
   - 在当前知识条目规模下，足够稳定，维护成本也更低。
   - 暂时没有必要为了“看起来更先进”而强行引入 pgvector、Milvus 或复杂重排序链路。

4. **适合作为内网先推广的培训产品**
   - 对于咨询员培训，系统的价值在于：
     - 快速熟悉常见患者问题
     - 统一高质量回复标准
     - 降低老员工一对一带教成本
     - 把“优秀对话”转为可持续复用的知识资产

#### 12.29.6 当前系统最适合的使用方式

结合当前功能完成度，推荐按以下顺序使用系统：

##### 场景 A：新入职咨询员入职培训（优先级最高）

##### 场景 B：老员工快速查话术

### 12.31 AI 回复速度优化：高置信直通机制（2026-04-28 实施）

#### 12.31.1 背景与问题

原有 Copilot 每次必须调用 LLM（`qwen-turbo` 约 1-3 秒，`qwen-plus` 约 2-4 秒），  
对于知识库命中质量很高的问题，这段 LLM 等待时间是多余的，用户感知为"卡顿"。

#### 12.31.2 优化方案：高置信直通（High-Confidence Direct Pass）

在 `faq_llm.py` 的 `copilot_answer` / `copilot_answer_stream` 两个函数前端加入直通判断：

```python
HIGH_CONF_DIRECT_THRESHOLD = 0.82

def _try_direct_faq_answer(faq_context, quality_mode=None) -> CopilotResult | None:
    # 顶部匹配相似度 >= 0.82 → 直接返回 best_answer，跳过 LLM
    # quality 模式除外（用户明确要求 LLM 润色）
```

**触发条件**：
- 语义搜索排名第一的 FAQ 相似度 `≥ 0.82`
- 质量模式不为 `quality`（auto / fast / balanced 均触发）
- 该 FAQ 的 `best_answer` 字段非空

**触发后效果**：
- 响应时间从 **1-3 秒** 降至 **<100ms**（仅含 Embedding API 时间）
- `note` 字段返回 `"high_conf_direct"`，前端可展示"直接匹配"标识
- `confidence` 字段等于原始相似度分数

#### 12.31.3 适用范围

| Copilot 端点 | 是否生效 |
|---|---|
| `POST /faq/copilot`（非流式） | ✅ |
| `POST /faq/copilot/panel`（快捷回复面板） | ✅ |
| `POST /faq/copilot/stream`（SSE 流式） | ✅（一次性输出 best_answer） |
| `quality` 模式 | ❌（强制走 LLM 润色，不触发直通） |

#### 12.31.4 效果预估

假设知识库已积累 200+ 条高质量 FAQ，常见问题匹配率约 60-70%：
- 约 **60-70% 的查询**实现秒回（≤100ms）
- 约 **20-30% 的查询**匹配中等相似度，走 LLM 快速路径（turbo 约 1-2s）
- 约 **5-10% 的查询**为罕见/特殊问题，走 LLM 深度路径（balanced/quality 约 2-4s）

#### 12.31.5 阈值调整方式

通过修改 `faq_llm.py` 顶部常量：
```python
HIGH_CONF_DIRECT_THRESHOLD = 0.82  # 可调整，建议范围 0.75 - 0.90
```
- 降低阈值（如 0.75）：更多查询触发直通，速度更快，但部分回复可能不够精确
- 提高阈值（如 0.90）：仅极高置信时触发，安全但减少直通比例

#### 12.31.6 模型配置优化（2026-04-28 同步调整）

原 `qwen3-max` 是阿里云通义千问中最贵的旗舰模型（¥0.04/千 tokens），对本系统属于过度配置。  
结合高置信直通后，推荐的最优性价比搭配如下：

| 用途 | 原配置 | 新配置 | 费用变化 |
|------|--------|--------|----------|
| FAQ 管道（批量低频）+ quality 模式 | `qwen3-max` ¥0.04/千 | `qwen-plus` ¥0.004/千 | **降低 10 倍** |
| Copilot auto / balanced 模式 | `qwen-plus` ¥0.004/千 | `qwen-turbo` ¥0.003/千 | 降低 25% |
| Copilot fast 模式 | `qwen-turbo` ¥0.003/千 | `qwen-turbo`（不变） | — |
| 高置信直通（≥0.82）| LLM 调用 | **直接返回，0 费用** | **约 60-70% 查询免费** |

**调整后 `config.py` 默认值**：
```python
dashscope_model = "qwen-plus"       # 管道 + quality 模式
dashscope_copilot_model = "qwen-turbo"           # fast 模式
dashscope_copilot_balanced_model = "qwen-turbo"  # auto / balanced 模式
```

> 如需恢复最高质量，在 `.env` 里覆盖 `DASHSCOPE_MODEL=qwen3-max` 即可，代码逻辑无需改动。
